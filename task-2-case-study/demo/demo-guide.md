# Task 2 - Demo Guide

## Purpose of the demonstration

The classroom demonstration should not be framed as a magic trick in which AI solves a task instantly. It should be framed as a compact demonstration of what AI-assisted SWE actually looks like in practice inside a repository workflow. The audience should leave with a clear understanding that the tool can accelerate parts of the process, but that engineering judgment remains necessary.

## Suggested demo narrative

Begin by showing the repository and the selected issue. Explain the expected behavior and identify the part of the code that appears relevant. Then show one concrete moment where GitHub Copilot helps, for example by proposing a draft implementation or a test. After that, show the critical engineering step: the suggestion is reviewed rather than accepted blindly. Finally, run the validation and explain what still required human intervention.

This demo structure is strong because it mirrors the main conclusion of the literature. AI assistance is helpful, but its value depends on bounded tasks, repository context, and careful human oversight [1], [2], [11].

## What the audience should learn

The most important learning outcome is that AI-assisted SWE changes the workflow more than it eliminates it. Drafting becomes faster, but reviewing and validating remain central. The second learning outcome is that the technology is most effective when the task is well framed and the repository offers enough context. The third is that the main trade-offs are not abstract. They are visible directly in the coding session: speed, uncertainty, context, and verification.
