# Task 1.2 - Motivations

## Why this topic is important now

The strongest motivation for studying AI-assisted SWE is that software development has become bottlenecked by complexity rather than by syntax. Mature systems contain accumulated abstractions, historical decisions, tightly coupled modules, inconsistent documentation, and non-trivial validation processes. A developer working in such a system spends substantial time searching, understanding, checking, and coordinating. AI tools are relevant because they promise to compress some of that friction. The best current evidence supports the view that AI can reduce effort on bounded tasks. In the GitHub Copilot controlled study, participants with access to Copilot completed the target task 55.8% faster [1]. In GitHub’s enterprise study with Accenture, developers reported lower mental effort on repetitive tasks and less time spent searching for examples, while telemetry also showed more pull requests and more successful builds [3]. This makes AI-assisted SWE worth studying not as hype, but as an engineering intervention with observable effects.

A second motivation is that the adoption curve is unusually steep. GitHub’s 2024 and 2025 Octoverse reports show rapid growth both in public AI-related development activity and in AI usage by developers themselves [7], [8]. This matters because topics in ASE should ideally be both technically meaningful and representative of current industry practice. AI-assisted development now satisfies both conditions. It is no longer a speculative future trend. It is already shaping the workflow, language choices, and tooling expectations of software teams.

A third motivation is that the benefits are uneven and context dependent. AI tools often perform well on repetitive, local, or well-scaffolded tasks, but they are less reliable when hidden context, architectural intent, or cross-file reasoning become important. Cisco researchers studying real-world use of GitHub Copilot reported up to 50% time savings for documentation and autocompletion and around 30% to 40% savings in repetitive coding, unit test generation, debugging, and pair programming tasks, but also documented weaker performance on complex tasks, large functions, multiple files, and proprietary contexts [11]. This asymmetry is precisely why the topic is interesting. If AI only helped everywhere, the case would be trivial. If it helped nowhere, it would also be trivial. The engineering problem exists because the benefits are real but incomplete.

A fourth motivation is that AI changes the distribution of effort inside the development process. If a developer obtains a draft implementation more quickly, the time saved does not simply disappear. Instead, part of the work shifts toward review, verification, test adequacy, and context checking. This is consistent with research on ChatGPT answers to Stack Overflow questions, where 52% of answers contained incorrect information and 77% were verbose, yet users still preferred many of those answers because they were comprehensive and well written [2]. The implication for software engineering is important: an articulate suggestion can be persuasive without being reliable. This makes AI-assisted SWE a useful topic for discussing trust calibration, human oversight, and evaluation.

A fifth motivation is organizational. Deloitte’s 2026 software outlook argues that software companies are moving toward AI-first development models, expecting productivity gains across the SDLC while also redesigning team structures and talent models [9]. Gartner adds that developers will increasingly move from direct implementation toward orchestration and higher-level problem solving [5]. These claims connect the topic to workforce design, learning, mentorship, and governance. In other words, AI-assisted SWE is not only a tooling question. It is a question about how software engineering as a discipline is being reconfigured.

## Main challenges and topics discussed in the literature

The literature converges on several recurring themes. The first is **productivity and flow**. AI tools can reduce friction in code drafting, boilerplate generation, routine test writing, and navigation of familiar patterns [1], [3], [11]. The second is **correctness and over-trust**. Outputs can be plausible yet wrong, especially when users do not inspect them critically [2]. The third is **context dependence**. Benefits tend to be strongest where the problem is local and well framed, and weaker where the task depends on architecture, hidden constraints, or proprietary knowledge [11]. The fourth is **security and governance**. Public AI systems can create data leakage risks, as the Samsung incident demonstrated [10]. The fifth is **organizational transformation**. AI changes what skills matter, how teams are structured, and how productivity is measured [5], [9].

These themes motivate the project because they show that the real challenge is not whether AI can generate code at all. The real challenge is whether AI can be integrated into software engineering practice in a way that improves outcomes without degrading quality, security, or collective understanding. That is a substantially deeper and more relevant question for ASE.
