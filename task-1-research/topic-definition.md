# Task 1.1 - Topic Definition

## AI-assisted Software Engineering as an ASE challenge

This project focuses on **AI-assisted software engineering (AI-assisted SWE)**. In the context of ASE, the topic is not simply about whether an AI model can write code. The more relevant question is how AI changes software engineering work inside **large, interdependent, evolving systems**. These systems contain many files, architectural layers, hidden assumptions, team conventions, review processes, and operational constraints. In such an environment, any tool that accelerates one activity can create pressure somewhere else, for example in review, validation, integration, security, or governance. That is why AI-assisted SWE is a strong fit for a course on large complex software systems.

A useful working definition is the following. AI-assisted SWE refers to the use of generative AI systems to support or partially automate software engineering tasks such as code generation, code explanation, test creation, documentation, debugging, refactoring, requirements support, and workflow orchestration. The keyword in this definition is **assisted**. The technology is relevant because it augments software engineering work, but the engineering responsibility still remains with the human team. The literature currently supports this framing better than stronger claims about full automation. In a controlled experiment, developers using GitHub Copilot completed a programming task 55.8% faster than the control group, which shows genuine productivity potential, but the result does not imply that review, design, or accountability disappear [1].

The topic also matters because the technology is no longer peripheral. GitHub’s Octoverse 2024 report states that GitHub hosted about 137,000 public generative AI projects, with 98% year-over-year growth, while the 2025 Octoverse report states that more than 1.1 million public repositories now use an LLM SDK and that 80% of new developers on GitHub use Copilot in their first week [7], [8]. These numbers matter for two reasons. First, they show that AI is becoming part of the mainstream developer toolchain. Second, they indicate that the conversation has moved beyond curiosity and into real engineering practice.

A second distinction is needed between **assistant-style** and **agent-style** AI. Assistant-style tools operate inside a local workflow and help the user with bounded tasks such as writing a function, proposing a test, or explaining a file. Agent-style systems operate at a higher level of autonomy and attempt multi-step execution, such as planning a fix, editing multiple files, running validation, and proposing a patch. Gartner argues that the direction of travel in software engineering is from isolated code assistance toward broader AI-native engineering and agentic workflows [5], [6]. However, Gartner also emphasizes that teams must balance automation with human oversight depending on workflow complexity, business criticality, and risk [5]. This is particularly relevant in large systems, where local code suggestions can look plausible even when they conflict with architecture, domain assumptions, or operational constraints.

For the purposes of this project, the topic is therefore defined more precisely as follows: **AI-assisted SWE is the use of generative AI tools to support software engineering tasks in a way that changes productivity, quality, coordination, and governance in large software systems.** This definition is deliberately broader than code completion and narrower than general AI adoption. It allows us to study both the technical benefits and the engineering trade-offs.

## Why this topic belongs in a project on large complex systems

Large complex systems are difficult because the main challenge is rarely typing code. The main challenge is understanding interactions. Developers must reason about multiple modules, interfaces, runtime behavior, tests, deployment pipelines, security controls, technical debt, and team conventions. In that setting, a tool that accelerates local code writing can be valuable, but only if it does not degrade correctness, maintainability, or coordination. Research on GitHub Copilot in large proprietary projects reports tangible efficiency gains, but also highlights that the tool struggles with complex tasks, large functions, multiple files, and proprietary or highly specific contexts [11]. This aligns well with the concerns of ASE: local help is not enough when the real problem is systemic complexity.

The topic is also timely from an industry perspective. Gartner predicts that by 2028, 90% of enterprise software engineers will use AI code assistants, up from less than 14% in early 2024, and that developer work will shift from implementation toward orchestration, system design, and ensuring that AI-generated outputs are fit for purpose [5]. Deloitte similarly argues that software is being created faster and cheaper, that firms are moving toward AI-first engineering and product design, and that development teams are likely to be redesigned into smaller, AI-augmented structures [9]. These are not small cosmetic changes. They suggest that AI-assisted SWE affects not only how code is written, but how teams are staffed, trained, measured, and governed.

In summary, AI-assisted SWE is an ASE topic because it sits exactly at the intersection of technical productivity and socio-technical complexity. It promises speed, but forces new questions about trust, review, architecture, security, learning, and responsibility. Those questions are much closer to the real challenge of large systems than the simplistic claim that “AI writes code for developers.”
